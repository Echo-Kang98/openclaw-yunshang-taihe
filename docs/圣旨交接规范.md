# 三省六部 · 圣旨交接规范

> 基于 Anthropic Long-Running Agent 最佳实践，为云上泰和三省六部体系设计的任务交接机制。
> 每次任务完成后，执行 agent 必须撰写交接文档，确保下一任执行者（或下一 session）能快速上手。

---

## 一、交接文档格式

每个任务完成后，执行 agent 必须生成如下格式的交接文档：

```markdown
# 圣旨交接文档

**任务ID**: JJC-YYYYMMDD-NNN
**任务名称**: xxx
**执行部门**: [部门名]
**执行人**: [agent名]
**完成时间**: YYYY-MM-DD HH:MM
**交接时间**: YYYY-MM-DD HH:MM

---

## 一、本次完成情况

### 已完成 ✅
- [具体事项1]
- [具体事项2]

### 未完成项 ❌
- [未完成事项] — 原因：[原因] — 阻塞：[阻塞项]

---

## 二、关键产物

| 产物 | 路径/位置 | 说明 |
|------|-----------|------|
| 代码/文件 | `/path/to/file` | xxx |
| 文档 | `/path/to/doc` | xxx |
| 数据库 | Notion/表名 | xxx |

---

## 三、给下一任的信息

### 上下文延续
- [继续这项工作需要知道什么]

### 风险提醒
- [已知的风险或未解决的问题]

### 下一步行动
1. [第一步]
2. [第二步]
3. [第三步]

---

## 四、验证方式

- [ ] 如何验证这项工作已完成
- [ ] 测试方法（如有）

---

## 五、联系人（若需要人工介入）

- [部门/联系人] — [具体事项]
```

---

## 二、交接文档存储位置

| 场景 | 存储位置 |
|------|----------|
| 项目内任务 | `yunshang-taihe/docs/handover/JJC-YYYYMMDD-NNN.md` |
| 即时任务 | 同步到 Notion 旨意库（结果字段） |
| 紧急交接 | 飞书消息直接通知下一任 |

---

## 三、各部门交接责任

| 部门 | 职责 |
|------|------|
| 尚书省 | 派发任务时必须指定"交接给谁" |
| 执行部门 | 完成时必须撰写交接文档 |
| 太子（中转） | 转发任务时附上前任交接文档摘要 |
| 门下省 | 审核时检查交接文档完整性 |

---

## 四、新 session 启动标准动作

> 每次 agent 新 session 开始时，必须按以下顺序执行：

```
1. 读取 MEMORY.md（了解皇上基本信息）
2. 读取 memory/YYYY-MM-DD.md（了解今日事件）
3. 检查看板：JJC 任务中是否有进行中的任务
4. 如有进行中任务 → 读取对应交接文档
5. 向太子（taizi）确认上下文是否需要补充
```

---

## 五、交接质量标准

**合格交接必须包含**：
- ✅ 本次完成的具体产出（不能只说"做了"）
- ✅ 未完成项及原因（不能假装完成了）
- ✅ 下一步行动（明确到可执行）
- ✅ 风险提醒（如有）

**不合格交接**：
- ❌ 只写"已完成"
- ❌ 没说未完成项
- ❌ 没有下一步
- ❌ 把烂摊子留给下一任
